Réglementation

Cadre réglementaire DORA

Digital Operational Resilience Act – Règlement UE 2022/2554 – En vigueur depuis janvier 2025
Art. 9 & 12
Protection & Continuité
  • Politique documentée de sauvegarde (RPO/RTO formalisés)
  • Tests périodiques obligatoires des procédures de restauration
  • Chiffrement des données en transit et au repos
  • Ségrégation des sauvegardes (isolées du réseau de production)
Art. 11
Politiques de rétention
  • Rétention minimale 7 ans pour les établissements financiers
  • Immutabilité des sauvegardes (protection anti-ransomware)
  • Traçabilité des accès aux données sauvegardées (audit log)
  • Support des supports longue durée (LTO WORM, stockage immuable)
Art. 13 & 25
Tests de résilience
  • TLPT (Threat Led Penetration Testing) annuel obligatoire
  • Tests de restauration documentés et validés trimestriellement
  • BCP/DRP incluant les workloads Kubernetes
  • Simulations de sinistre incluant perte totale d'un cluster
Art. 28-44
Risque tiers & reporting
  • Due diligence sur les fournisseurs de solutions de backup
  • Contrats TPSP avec SLA backup formalisés
  • Notification incidents ICT < 4h (initial), < 24h (intermédiaire)
  • Registre des dépendances critiques tiers

Veritas Technologies

Veritas NetBackup for Kubernetes

Solution enterprise – Opérateur natif Kubernetes – Intégration Data Domain & LTO
Déploiement & Intégration K8s
  • Opérateur Kubernetes natif (Helm chart + CRDs)
  • Support OpenShift 4.x certifié Red Hat
  • RBAC granulaire par namespace
  • Découverte automatique namespaces & PVCs
  • API REST pour orchestration et automatisation
Capacités de sauvegarde
  • Backup au niveau namespace, label selector, application
  • Cohérence application-aware (hooks pre/post)
  • Support snapshots CSI (Container Storage Interface)
  • Sauvegarde incrémentale block-level (BLIB) pour PVCs
  • Copie de catalogue chiffrée pour reprise catalogue
Intégration stockage LTO & Data Domain
  • Data Domain via OST (OpenStorage Technology) — DDBoost natif
  • LTO via Media Server — support WORM, chiffrement AES-256
  • CloudCatalyst pour staging vers S3 avant LTO
  • MSDP (Media Server Deduplication Pool) en alternative DD
  • Multi-target DD + LTO simultanément via SLP
Reporting & Compliance
  • OpsCenter : tableau de bord centralisé multi-cluster
  • Rapports conformité DORA (rétention, chiffrement, tests)
  • Audit log complet sauvegarde / restauration
  • Alertes proactives (échec job, quota, expiration)
Veritas NetBackup

Analyse Points Forts / Points Faibles

Évaluation objective basée sur les fonctionnalités produit et retours terrain
✔ Points forts
  • Maturité produit : +30 ans d'expérience enterprise, référence secteur financier
  • Intégration LTO la plus complète : SLP gère automatiquement le tiering DD → LTO
  • DDBoost natif : déduplication côté client, réduction trafic réseau 50–80%
  • Granularité fine : restauration fichier, namespace, PVC, manifeste individuel
  • Multi-cloud : AWS S3, Azure Blob, GCS + LTO (flexibilité cible)
  • Chiffrement AES-256 en transit et au repos (exigence DORA Art. 9)
  • SLA & support enterprise : 24/7, certifications FIPS 140-2
  • Catalogue répliqué : survie à la perte complète du master server
✘ Points faibles
  • Complexité de déploiement : architecture multi-composants (Master, Media Servers, Client)
  • Coût de licence élevé : modèle FETB + options coûteuses (Kubernetes Add-on, CloudCatalyst)
  • Interface utilisateur vieillissante : Web UI améliorée mais inférieure aux concurrents cloud-native
  • Agilité Kubernetes limitée : conçu pour VMs/physique, support K8s en add-on
  • Footprint important : consommation ressources significative dans le cluster
  • Cycles de release longs comparés aux outils cloud-native
  • Courbe d'apprentissage élevée pour les équipes DevOps
Positionnement : Solution de référence pour les environnements enterprise hybrides nécessitant une intégration LTO et Data Domain éprouvée. Recommandée pour les DSI matures avec équipes backup dédiées.
Commvault Systems

Commvault Complete Backup & Recovery for Kubernetes

Plateforme unifiée – Protection données exhaustive – DDBoost natif
Déploiement & Intégration K8s
  • Opérateur Kubernetes via Helm chart (déploiement simplifié)
  • Certifié Red Hat OpenShift 4.x (catalogue OperatorHub)
  • MediaAgent déployable comme pod dans le cluster
  • Support multi-cluster depuis une console unique (Command Center)
  • RBAC natif, intégration LDAP/AD/SAML
Capacités de sauvegarde
  • Granularité : cluster, namespace, application, pod, PVC, fichier
  • IntelliSnap : snapshots CSI (NetApp, Pure Storage, Dell EMC…)
  • Sauvegarde incrémentale forever (CBT/CBR pour volumes persistants)
  • Hooks applicatifs pre/post pour cohérence applicative
  • Live Sync : réplication quasi-temps-réel DR (RPO < 15 min possible)
Intégration stockage LTO & Data Domain
  • DDBoost intégré nativement dans le MediaAgent Commvault
  • LTO via MediaAgent : support bibliothèques physiques et VTL
  • Auxiliaire Copy automatique : Data Domain → LTO selon rétention
  • WORM sur LTO avec cartouches WORM + chiffrement matériel LTO-9
  • Air-gap automatique : isolation réseau des sauvegardes sur bande
Compliance & Reporting
  • Metallic (SaaS) : option cloud pour copies hors-site
  • Compliance Lock : immuabilité des sauvegardes (anti-ransomware)
  • Rapports DORA prêts à l'emploi (rétention, chiffrement, tests)
  • Threat Scan : analyse malware des sauvegardes avant restauration
Commvault

Analyse Points Forts / Points Faibles

Évaluation objective basée sur les fonctionnalités produit et retours terrain
✔ Points forts
  • Interface Command Center moderne et intuitive : adoption rapide
  • Plateforme unifiée : backup VMs, physique, K8s, SaaS (M365, Salesforce)
  • IntelliSnap : intégration profonde avec les baies storage enterprise
  • Compliance Lock natif : immuabilité WORM sans matériel dédié
  • Threat Scan : détection malware/ransomware dans les backups avant restauration
  • Live Sync : DR actif-passif pour applications critiques K8s (faible RPO)
  • Metallic (SaaS) : copies offsite sans infrastructure supplémentaire
  • Coût total maîtrisé : licences par capacité, options claires
✘ Points faibles
  • Architecture complexe : CommServe, MediaAgents, clients — nombreux composants
  • Footprint ressource : MediaAgent dans le cluster consomme des ressources
  • Support K8s moins mature que Portworx sur certaines fonctionnalités avancées
  • Licensing complexe : multiplication des modules optionnels — coût difficile à anticiper
  • Intégration LTO via MediaAgent uniquement : pas de support direct dans le cluster
  • Restauration granulaire fichier plus lente que les solutions snapshots natives
Positionnement : Solution polyvalente idéale pour les DSI souhaitant une plateforme unifiée couvrant l'ensemble des workloads. Excellent équilibre fonctionnalités / complexité / TCO.
Pure Storage

Portworx PX-Backup

Solution cloud-native Kubernetes-first – Backup & DR orientés conteneurs
Déploiement & Intégration K8s
  • Architecture 100% Kubernetes-native : déployé entièrement comme pods
  • PX-Central : console de gestion multi-cluster cloud-native
  • OpenShift certifié, intégration OperatorHub
  • Stockage distribué Portworx Enterprise : CSI driver pour PVCs HA
  • Intégration Stork (Storage Operator for Kubernetes)
Capacités de sauvegarde
  • PX-Backup : backup namespace/application avec cohérence 3DSnap
  • AsyncDR : réplication asynchrone inter-cluster pour disaster recovery
  • Granularité : namespace, application, PVC individuel
  • Hooks pre/post configurables (scripts de quiescence applicative)
  • Intégration Velero : compatible avec l'écosystème open-source
⚠ Intégration stockage (ATTENTION)
  • Cible backup native : S3 compatible (MinIO, AWS S3, Azure Blob, GCS)
  • Data Domain : intégration via NFS/CIFS uniquement — SANS DDBoost natif
  • LTO : NON supporté nativement — nécessite passerelle VTL ou solution tierce
  • Recommandation LTO : pipeline externe requis (Veeam, script dédié)
Points de différenciation
  • Autopilot : gestion automatique des règles de rétention
  • PX-DR : RPO quasi-nul pour applications stateful (réplication synchrone)
  • Intégration ArgoCD / Flux : GitOps-native pour restaurations déclaratives
  • Chiffrement per-volume : granulaire au niveau volume (clés par namespace)
Portworx PX-Backup

Analyse Points Forts / Points Faibles

Évaluation objective basée sur les fonctionnalités produit et retours terrain
✔ Points forts
  • Architecture cloud-native pure : aucun agent lourd, déploiement comme tout workload K8s
  • 3DSnap : cohérence applicative au niveau stockage distribué sans agent dans les pods
  • RPO très faible : PX-DR réplication quasi-synchrone pour DR inter-cluster
  • Flexibilité des cibles : tout objet store S3 compatible (cloud ou on-prem)
  • Intégration GitOps : restauration déclarative via ArgoCD/Flux
  • PX-Central : gestion multi-cluster simple et tableau de bord clair
  • Orientation DevOps : API-first, populaire auprès des équipes containers modernes
  • Chiffrement granulaire : par volume et par namespace
✘ Points faibles
  • ⚠ LTO non supporté nativement : POINT BLOQUANT pour rétention 10 ans DORA — nécessite architecture tierce complexe
  • ⚠ Data Domain sans DDBoost : NFS uniquement — perte des optimisations de déduplication distribuée
  • Couplage fort Portworx Storage : optimal uniquement avec Portworx Enterprise (vendor lock-in)
  • Immaturité compliance enterprise : rapports DORA moins riches, personnalisation limitée
  • Support LTO absent du roadmap officiel (focus 100% cloud/objet store)
  • Périmètre K8s uniquement : ne couvre pas VMs, physique ou workloads legacy
⚠ Avertissement : L'absence de support natif LTO est un point BLOQUANT dans le contexte de cette étude (rétention 10 ans DORA). Portworx peut être retenu uniquement si une solution complémentaire de tiering LTO est mise en place en parallèle.

Recommandation & Plan d'action

Synthèse de l'analyse comparative – Choix recommandé selon les contraintes DORA
🏆 Recommandation principale

Commvault Complete for Kubernetes — Score 8.9 / 10

Meilleur équilibre fonctionnalités / conformité DORA / TCO. Intégration LTO et DDBoost nativement. Compliance Lock anti-ransomware. Plateforme unifiée VMs + K8s + SaaS. Rapports DORA prêts à l'emploi.

🥈 Alternative : Veritas NetBackup (8.6/10)

Recommandé si l'équipe dispose d'expertise NetBackup existante ou si l'environnement est très hétérogène (nombreux systèmes legacy). Intégration LTO la plus mature du marché.

⚠ Portworx : Usage spécifique uniquement (5.9/10)

À retenir uniquement pour workloads K8s pure cloud sans contrainte LTO. Non recommandé dans ce contexte DORA sans architecture hybride complémentaire.

Plan d'action recommandé
Phase 1 – J0 à J30
  • Cadrer les exigences RPO/RTO par namespace/application avec les équipes métier
  • Inventorier les PVCs et volumes à protéger (volumétrie, fréquence)
  • Réaliser un PoC Commvault sur un cluster de qualification OpenShift
Phase 2 – J30 à J90
  • Déployer l'opérateur Kubernetes en production (namespace par namespace)
  • Configurer les cibles Data Domain (DDBoost) et SLA par tier
  • Implémenter les règles de tiering automatique DD → LTO selon rétention DORA
Phase 3 – J90 à J180
  • Tester et documenter les procédures de restauration (Art. 13 DORA)
  • Former les équipes Ops, rédiger les runbooks de restauration d'urgence
  • Produire les premiers rapports DORA et planifier les audits périodiques

Conclusion

Étude comparative Backup Kubernetes / Red Hat OpenShift
  • 1
    Commvault s'impose comme la solution la mieux adaptée : intégration DDBoost et LTO native, Compliance Lock, rapports DORA, et plateforme unifiée.
  • 2
    NetBackup reste une alternative solide pour les environnements disposant d'une expertise existante et d'architectures très hétérogènes.
  • 3
    Portworx est disqualifié dans ce contexte précis en raison de l'absence de support LTO natif, condition sine qua non de la rétention 10 ans DORA.
  • 4
    L'intégration Data Domain via DDBoost est un pré-requis fonctionnel — NetBackup et Commvault la supportent nativement ; Portworx ne la supporte pas.
  • 5
    La conformité DORA impose des tests de restauration documentés, une immuabilité WORM et un chiffrement AES-256 — couverts par les deux solutions retenues.
